# Haerbin Retrieval：V1 → V2 → V3 完整演进说明

本文按版本说明 Haerbin 在 TREC RAG 2026 Retrieval Task 上做过的工作：每个版本解决什么问题、增加了哪些模块、如何实现、对应哪些文件、指标发生了什么变化，以及我们从实验中得到了什么结论。

## 0. 先明确任务边界

官方已经提供：

- ClimbMix 文档集合及其 Pyserini/Anserini BM25 索引；
- 远程 Pyserini REST API；
- 22 个开发 topics 和 119 个正式 test topics；
- 三套开发集 UMBRELA qrels。

因此本项目没有重新下载全库、重新切 chunk、给全库生成 embedding 或新建向量数据库。我们的工作分成两部分：

1. **候选生成**：改变查询方式，让官方 BM25 API 找到更多不同文档；
2. **候选排序**：对已经召回的文档使用语义模型重新排序。

这一区别非常重要：reranker 能改善 nDCG/MAP，但不能找回从未进入候选池的文档。

## 1. 版本总览

| 版本 | 核心目标 | 主要新增内容 | 代表结果 |
| --- | --- | --- | --- |
| V1 | 先跑通并建立可迭代 baseline | 官方 BM25、格式校验、查询改写、多查询、RRF、初版 MiniLM/BGE | 多查询和语义重排有效，但只处理 Top-1000 |
| V2 | 扩大深层召回并建立稳定主线 | Top-5000、多级 cross-encoder、原排名保护、NIST 评测 | nDCG@10 `0.7210`，Recall@5000 `0.5225` |
| V3 | 加入更细粒度混合信号，精修头部排序 | dense hybrid、ColBERT、文档压缩、RankLLM | nDCG@10 `0.7243`，Recall@1000 `0.3423` |

最终主线：

```text
官方 BM25
→ DeepSeek 查询压缩/分解
→ weighted RRF Top-5000
→ MiniLM cross-encoder Top-5000
→ BGE v2-m3 cross-encoder Top-1000
→ BGE-small dense hybrid
→ ColBERTv2 MaxSim Top-100
→ query-aware 压缩 + DeepSeek RankLLM Top-20
→ 六列 TREC runfile
```

## 2. V1：从官方 BM25 到多查询语义检索 MVP

### 2.1 V1 要解决的问题

V1 的第一目标不是追求最高分，而是建立一条可以重复运行、验证和评测的最小闭环：

```text
读取 topic → 调官方 API → 保存响应 → 生成 runfile → 校验 → 计算指标
```

在 BM25 跑通后，我们发现复杂自然语言查询经常同时包含多个方面。把整段问题原样交给 BM25，容易出现词汇稀释和表达不匹配，因此 V1 又加入了查询改写、多查询检索和第一轮语义重排。

### 2.2 V1-A：官方 BM25 baseline

实现文件：

- `src/healthcheck.py`：确认 token、服务地址和索引可用，不打印 token；
- `src/retrieve.py`：读取 TSV topic，调用官方 `climbmix-400b` API；
- `src/validate_run.py`：检查六列 TREC 格式、rank 连续性、topic 和 docid；
- `src/evaluate_run.py`：使用公开 qrels 做本地快速评测。

具体实现：

1. 从 `data/topics/rag25-topics-dev.tsv` 读取 22 个开发查询；
2. 每个查询请求官方 BM25 Top-1000；
3. 将 API 原始 JSON 缓存在 `retrieval_mvp/cache/`；
4. 输出 `qid Q0 docid rank score run_id` 六列 runfile；
5. 用 validator 拒绝缺 topic、重复 docid、错误 rank 和非法数值。

这一阶段证明：我们不需要自己建立 ClimbMix 索引，也能生成合法 Retrieval submission。

### 2.3 V1-B：DeepSeek 查询改写

实现文件：

- `src/query_rewrite.py`；
- `src/retrieve_multiquery.py`。

每个 topic 被构造成：

```text
原始查询
压缩查询
5 个 aspect 子查询
```

DeepSeek 只负责构造检索查询，不直接判断文档相关性。输出经过以下校正：

- 保留原问题，避免改写完全偏离；
- 删除重复或空子查询；
- 限制子查询数量；
- LLM 失败时可回退到启发式拆分；
- 按 topic 缓存改写结果，避免反复付费调用。

### 2.4 V1-C：多查询检索和 weighted RRF

实现文件：`src/rrf.py`、`src/retrieve_multiquery.py`。

每条查询分别调用同一个官方 BM25 API，随后用 Reciprocal Rank Fusion 合并：

```text
RRF(d) = Σ weight_i / (k + rank_i(d))
```

我们给原始查询/BM25 更高权重，给压缩查询和子查询较低权重。这样既利用多种表达扩大候选，又避免某条 LLM 子查询主导整个结果。

### 2.5 V1-D：MiniLM 和 BGE 初版重排

实现文件：`src/rerank.py`。

初版使用 cross-encoder 直接评估查询和候选文档：

```text
[query, document] → cross-encoder → semantic score
```

我们分别测试：

- `cross-encoder/ms-marco-MiniLM-L-6-v2`；
- `BAAI/bge-reranker-v2-m3`；
- 纯语义排名；
- 原排名与语义排名按不同权重融合。

实验表明纯语义排序容易破坏 BM25 对实体、数字和专有词的精确匹配，因此后续版本都保留原排名保护。

### 2.6 V1 历史结果

V1 报告采用“22 topics、三套 qrels 逐 topic 指标平均、相关阈值 `>=2`”的早期诊断口径：

| V1 run | nDCG@10 | nDCG@20 | Recall@100 | Recall@1000 | MAP |
| --- | ---: | ---: | ---: | ---: | ---: |
| BM25 | 0.4601 | 0.4585 | **0.0997** | 0.2506 | 0.1386 |
| DeepSeek-RRF | 0.4971 | 0.4768 | 0.0974 | **0.2925** | 0.1615 |
| MiniLM-o4 | 0.5440 | 0.5174 | 0.0973 | **0.2925** | 0.1669 |
| BGE-o2 | **0.5736** | **0.5454** | 0.0947 | **0.2925** | 0.1666 |

V1 结论：

- 查询改写/RRF 确实扩大了已知相关文档覆盖；
- MiniLM/BGE 明显改善头部排序；
- Recall@100 有轻微下降，说明语义优化和浅层召回存在权衡；
- Top-1000 候选池限制了深层 Recall，成为 V2 的主要改进方向。

> 注意：V1 表格是三套 qrels 平均的历史诊断结果；后面的 V2/V3 主表采用 Codex qrels + NIST `trec_eval`。两张表用于各自版本内比较，绝对数值不能直接相减。

## 3. V2：Top-5000 深候选池与两级 Cross-Encoder

### 3.1 V2 要解决的问题

V1 证明了重排有效，但也暴露出一个事实：只在 Top-1000 内调排序，无法继续提高深层召回。

V2 的目标变成：

1. 把多查询候选池扩大到 Top-5000；
2. 用较快模型粗排全部 5000；
3. 用较强模型精排前 1000；
4. 保留完整 5000 深度，避免裁剪后丢失已召回文档。

### 3.2 V2-A：扩大候选池

`retrieve_multiquery.py` 的关键配置：

```text
原查询 + 压缩查询 + 5 个子查询
每条查询取 Top-1000
baseline weight = 4
RRF 合并并去重
输出 Top-5000
```

多个查询的结果会重叠，因此不是简单得到 7000 篇。去重和 RRF 后保留最多 5000 个候选。

### 3.3 V2-B：MiniLM 粗排 Top-5000

MiniLM 的优点是速度和显存开销较低，适合先处理完整候选池。它的任务不是最终决定前10名，而是把更可能相关的文档送入下一阶段。

主要参数：

```text
depth = 5000
original weight = 4
semantic weight = 1
```

模型分数写入 `cache/reranker/`。调整融合权重时可以直接读取缓存，不需要重新跑 GPU 推理。

### 3.4 V2-C：BGE v2-m3 精排 Top-1000

MiniLM 之后，用更强的 BGE cross-encoder 对前1000进行精排。

我们保留了两种版本：

- **Balanced v2**：`original weight = 2`，更重视 nDCG；
- **Recall v2**：`original weight = 8`，更保护原检索顺序和 MAP/Recall。

最终采用 Balanced v2 作为后续 V3 的主输入，因为它的 Top-10 排序质量更好。

### 3.5 V2-D：统一到 NIST 评测

实现文件：

- `src/official_eval.py`：调用 NIST `trec_eval`；
- `src/analyze_runs.py`：批量比较多个 run；
- `src/fuse_runs.py`：对多个 run 做统一 weighted RRF。

本地 evaluator 改为与 NIST 一致的 linear-gain nDCG，并用 `trec_eval` 交叉核对。此前出现过的 BM25 `0.4140`、三套平均 `0.4601` 和当前 `0.5940` 来自不同阶段/不同口径；从 V2 开始，主结果统一采用：

```text
Codex UMBRELA qrels
relevance threshold >= 2
NIST trec_eval
graded linear-gain nDCG
```

### 3.6 V2 结果

| Run | nDCG@10 | nDCG@20 | Recall@100 | Recall@1000 | Recall@5000 | MAP |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 官方 BM25 | 0.5940 | 0.5881 | **0.1081** | 0.2664 | 0.2664 | 0.1381 |
| 旧 Balanced | 0.6902 | 0.6535 | 0.1036 | 0.3063 | — | 0.1621 |
| **Balanced v2 deep** | **0.7210** | **0.6655** | 0.0996 | **0.3381** | **0.5225** | 0.1913 |
| **Recall v2 deep** | 0.6792 | 0.6406 | 0.1059 | **0.3381** | **0.5225** | **0.1966** |

V2 相对 BM25 的主要成果：

- Balanced v2 nDCG@10：`+0.1270`；
- Recall@1000：`+0.0717`；
- Recall@5000：`+0.2561`；
- Balanced v2 MAP：`+0.0532`。

V2 结论：

- 当前最大收益来自候选池扩展和 cross-encoder，而不是更复杂的 LLM 排序；
- 保留 Top-5000 很重要，裁成 Top-1000 会让 Recall@5000 退化为 Recall@1000；
- Balanced 与 Recall-safe 参数确实存在取舍，不能用一组权重同时最大化所有指标；
- Recall@100 仍略低于 BM25，说明浅层召回没有随深层召回一起改善。

## 4. V3：Dense、ColBERT 与 RankLLM 头部精修

### 4.1 V3 要解决的问题

V2 已经显著改善了主指标。V3 不再大改候选生成，而是测试这些技术能否继续提供互补信号：

- 稀疏 + 稠密混合检索；
- token-level late interaction；
- LLM listwise 排序；
- 长文档压缩和排序输出校正。

### 4.2 V3-A：SQLite 文档库

实现文件：`src/build_docstore.py`。

官方 API 响应中已经包含候选正文。该脚本把不同查询缓存中的 `docid → text` 去重后写入本地 SQLite，方便 dense、ColBERT、RankLLM 和导出脚本复用正文。

这一步没有重新下载全库，只整理已经召回的候选正文。

### 4.3 V3-B：BGE-small dense hybrid

实现文件：`src/dense_rerank.py`。

使用 bi-encoder 分别编码查询和候选文档，计算向量相似度，再把 dense 排名与 V2 排名做 weighted RRF。

```text
V2 sparse/cross-encoder ranking
            +
BGE-small dense ranking
            ↓
weighted RRF
```

首次对约 10.8 万个 query-document pair 编码约 24 分钟，之后直接读取缓存。

Dense 独立 run 的 Recall@1000/MAP 很好，但 nDCG@10 不如 Balanced v2。因此最终只给 dense 较小权重，作为互补信号而不是主排序器。

### 4.4 V3-C：ColBERTv2 Top-100

实现文件：`src/colbert_rerank.py`。

ColBERT保留查询和文档 token 的向量，通过 MaxSim 找到文档局部证据，适合处理长文档中只有少数句子相关的情况。

由于 token-level 打分成本更高，我们只校正 Top-100，并通过 RRF 保留原排名。当前 GPU 上一次 Top-100 打分约 19 秒。

### 4.5 V3-D：query-aware 文档压缩

实现文件：`src/rankllm_rerank.py`。

把长文档直接送给 LLM 成本高且容易被无关内容干扰，因此在 RankLLM 前增加相关句提取：

1. 将文档切成句子；
2. 根据查询词和句子匹配程度选择证据句；
3. 保持句子原始顺序；
4. 每篇文档最多保留约 900 字符。

这里的“压缩”不会改写证据，只从原文抽取更相关的句子。

### 4.6 V3-E：DeepSeek RankLLM Top-20

同样由 `src/rankllm_rerank.py` 实现。

把 Top-20 压缩文档一次性交给 DeepSeek，让模型输出候选编号顺序。随后进行严格校正：

- 过滤不在候选集合中的编号；
- 删除重复编号；
- 对缺失编号按原顺序补齐；
- 用 original weight 保护已有强排序；
- 按 topic 缓存 LLM 输出。

22 个 topics 首次调用约 8 分钟，缓存命中后约 1 秒。

### 4.7 V3 消融结果

| Run | nDCG@10 | nDCG@20 | Recall@100 | Recall@1000 | Recall@5000 | MAP |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 官方 BM25 | 0.5940 | 0.5881 | **0.1081** | 0.2664 | 0.2664 | 0.1381 |
| Balanced v2 | 0.7210 | 0.6655 | 0.0996 | 0.3381 | **0.5225** | 0.1913 |
| Dense hybrid 独立 | 0.6676 | 0.6380 | 0.1013 | **0.3450** | **0.5225** | **0.1962** |
| V2 + Dense | 0.7150 | **0.6680** | 0.1016 | 0.3423 | **0.5225** | **0.1938** |
| + ColBERT Top-100 | 0.7214 | 0.6651 | 0.1016 | 0.3423 | **0.5225** | 0.1936 |
| **+ RankLLM Top-20（最终 V3）** | **0.7243** | 0.6658 | 0.1016 | 0.3423 | **0.5225** | 0.1937 |

V3 结论：

- Dense 提供了一点深层 Recall/MAP 增益，但权重过高会降低 Top-10 质量；
- ColBERT 恢复并略微改善了头部排序；
- RankLLM 把 nDCG@10 从 `0.7214` 提高到 `0.7243`，属于小幅精修；
- V3 的大部分基础能力仍来自 V2，多层模型不是越多越好；
- 所有 V3 模块都只重排现有候选，因此 Recall@5000 保持 `0.5225`。

### 4.8 三套 qrels 稳定性

| qrels | V3 nDCG@10 | V3 nDCG@20 | Recall@1000 | Recall@5000 | MAP |
| --- | ---: | ---: | ---: | ---: | ---: |
| Codex | 0.7243 | 0.6658 | 0.3423 | 0.5225 | 0.1937 |
| Ministral | 0.8138 | 0.7637 | 0.3015 | 0.4781 | 0.2156 |
| Qwen | 0.7215 | 0.6936 | 0.3285 | 0.5148 | 0.1993 |

三个 judge 的绝对值不同，但都表明 V3 具有可用的头部排序和约 0.48–0.52 的 Top-5000 已知相关文档覆盖。

我们还使用 Castorini 官方 RAGDoll/UMBRELA 流程，对 V3 的 22×Top-10 共 220 个 query-document pair 做了真实 LLM judgment。

| 项目 | 结果 |
| --- | ---: |
| Completed | 220 / 220 |
| Parse failures | 0 |
| Top-10 nDCG | 0.9034 |
| Grade >= 2 | 141 / 220（64.09%） |
| Grade >= 1 | 218 / 220（99.09%） |
| Mean grade | 1.7773 / 3 |

该测试使用官方 RAGDoll prompt/runner 和 `deepseek/deepseek-v4-flash` judge。它只评价 V3 自己 Top-10 的内部顺序，不是官方 leaderboard 分数，不能用于计算可信的 Recall@1000/5000。完整限制见 `ragdoll_top10.md`。

## 7. 当前成果和下一步

当前已经完成：

- 从官方 API 到合法 runfile 的完整闭环；
- 查询构建、稀疏检索、多查询融合；
- cross-encoder、dense、ColBERT、RankLLM 多阶段排序；
- 缓存、断点复用、输出校正、单元测试；
- 本地指标、NIST `trec_eval` 和 RAGDoll 三类验证；
- 开发集最终 V3 nDCG@10 `0.7243`、Recall@5000 `0.5225`。

当前瓶颈不是再增加一个 reranker，而是约 47.75% 的 qrels 已知相关文档没有进入 Top-5000。下一步应优先：

1. 按 topic 统计相关文档总数和 Top-100/1000/5000 漏检；
2. 分析漏检文档的实体别名、时间、地点、术语和 query facet；
3. 设计差异更大的查询路由，而不是继续生成相似改写；
4. 在可能的情况下引入能够独立访问全库的另一种召回器；
5. 用 BM25/V2/V3 合并候选池做公平的 RAGDoll shallow-pool 对比。

## 8. 评测口径提醒

- V1 历史表：三套 qrels 平均，适合观察 V1 内部相对变化；
- V2/V3 主表：Codex qrels + NIST `trec_eval`，作为当前统一主口径；
- RAGDoll 表：仅判断 V3 Top-10，不能代表全库 Recall 或官方最终成绩；
- 119 个正式 test topics 没有公开 qrels，正式分数必须等待比赛方评测。

运行产生的 API 响应、模型缓存、runfile、候选正文、RAGDoll judgment 和 API key 均只保存在本地并被 `.gitignore` 排除。
